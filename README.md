# Rearc Data Quest

A fully automated data pipeline built on AWS that ingests BLS time-series CSV data and US population API data, stores them in S3, runs analytics, and reports results — all orchestrated with Terraform.

---

## Architecture

```
EventBridge Scheduler (daily)
        │
        ▼
┌───────────────────┐        ┌─────────────────────────┐
│   questLambda     │───────▶│  S3: rearc-quest-bucket  │
│  (ingestion)      │        │  ├── api/data.json        │
└───────────────────┘        │  ├── csv/pr.*.csv         │
                             │  └── config/metadata.json │
                             └──────────┬────────────────┘
                                        │ s3:ObjectCreated
                                        │ on api/data.json
                                        ▼
                             ┌─────────────────────┐
                             │    SQS: questQueue   │
                             └──────────┬──────────┘
                                        │ Event Source Mapping
                                        ▼
                             ┌───────────────────────┐
                             │  questLambdaReport     │
                             │  (analytics & report)  │
                             └───────────────────────┘
                                        │
                                        ▼
                              CloudWatch Logs (reports)
```

### Components

| Component | Name | Purpose |
|-----------|------|---------|
| S3 Bucket | `rearc-quest-bucket-aws` | Central data store for all ingested files |
| Lambda | `questLambda` | Daily ingestion: fetches BLS CSVs + population API, syncs to S3 |
| Lambda | `questLambdaReport` | Triggered by SQS; runs analytics and logs reports to CloudWatch |
| SQS Queue | `questQueue` | Decouples ingestion from analytics; receives S3 create notifications |
| EventBridge Scheduler | — | Triggers `questLambda` on a daily schedule |
| IAM Role | `questLambdaRole` | Shared execution role for both Lambda functions |

- Python 3.11
- [Terraform](https://developer.hashicorp.com/terraform/install) >= 1.x

## How to Deploy

### Step 1 — Build the Lambda packages

Run these from the **repository root**:

```bash
# questLambda (ingestion)
bash sh/buildLambda.sh

# questLambdaReport (analytics)
bash sh/buildLambdaReport.sh
```

What each script does:

| Script | What it does |
|--------|-------------|
| `buildLambda.sh` | Creates `lambda/` directory and copies `src/*.py` into it |
| `buildLambdaReport.sh` | Creates `lambdaReport/` directory, copies `src/analyse.py`, and installs `requirements.txt` dependencies into it (so pandas etc. are bundled in the zip) |

### Step 2 — Set AWS credentials

```bash
export AWS_ACCESS_KEY_ID=<your-key>
export AWS_SECRET_ACCESS_KEY=<your-secret>
export AWS_DEFAULT_REGION=ap-south-1
```

Or use a named profile:

```bash
export AWS_PROFILE=<your-profile>
```

### Step 3 — Initialise and apply Terraform

```bash
cd infra
terraform init
terraform plan
terraform apply
```

Terraform will create:

- `rearc-quest-bucket-aws` (S3 bucket)
- `questLambda` (ingestion Lambda)
- `questLambdaReport` (analytics Lambda)
- `questQueue` (SQS queue)
- S3 → SQS notification on `api/data.json` creation
- `questLambdaRole` IAM role with S3, CloudWatch Logs, and SQS permissions

### Step 4 — Set Lambda environment variables

After apply, set the following environment variables on both Lambda functions in the AWS console (or add them to `main.tf` as `environment` blocks):

| Variable | Value |
|----------|-------|
| `AWS_BUCKET` | `rearc-quest-bucket-aws` |
| `AWS_REGION` | `ap-south-1` |

## Configuration

Terraform input variables (can be overridden via `-var` or a `.tfvars` file):

| Variable | Default | Description |
|----------|---------|-------------|
| `aws_region` | `ap-south-1` | AWS region for all resources |
| `s3_notification_object_key_prefix` | `api/data.json` | S3 key prefix that triggers the SQS notification |
| `s3_notification_object_key_suffix` | `""` | Optional suffix filter; empty means prefix-only |

Example override:

```bash
terraform apply -var='aws_region=us-east-1'
```

## Data Pipeline — What Happens End to End

1. **EventBridge Scheduler** triggers `questLambda` once a day.
2. `questLambda` runs two ingestion tasks in sequence:
   - **API ingestion** (`extract_api.py`): fetches US annual population data from the [DataUSA Tesseract API](https://honolulu-api.datausa.io) and writes `api/data.json` to S3.
   - **CSV sync** (`extract_csv.py`): scrapes the [BLS PR time-series directory](https://download.bls.gov/pub/time.series/pr), compares against a `config/metadata.json` manifest stored in S3, uploads new/updated files as `csv/pr.*.csv`, and deletes files that have been removed from the source.
3. Writing `api/data.json` fires an **S3 ObjectCreated notification** to `questQueue`.
4. The SQS **event source mapping** invokes `questLambdaReport` for each message.
5. `questLambdaReport` (`analyse.py`) reads both files from S3 and logs three reports to **CloudWatch Logs**:
   - Mean and standard deviation of US annual population (2013–2018).
   - Per-`series_id` best year (year with the highest average value).
   - Merged view of `series_id = PRS30006032`, `period = Q01` with population for matching years.

## Notes

- The `.env` file (git-ignored) holds local AWS credentials for running scripts directly. Never commit it.
- The BLS endpoint requires a browser-like `User-Agent` header to avoid 403 errors — this is already set in `extract_csv.py` per [BLS data access policy](https://www.bls.gov/bls/pss.htm).
- Both Lambda functions share the `questLambdaRole` IAM role. The role has `AmazonS3FullAccess`, `CloudWatchLogsFullAccess`, and a scoped SQS consume policy on `questQueue`.
